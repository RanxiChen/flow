# Agent Collaboration Workflow

本文档记录当前仓库中“用户 + Codex + Kimi”的协作流程。它是当前生效规则，不是最终定稿；后续会随着真实任务推进持续修订。

## 目标

在不丢失项目掌控感的前提下，引入 Kimi 作为受控执行器，帮助完成一部分实现类任务，同时由 Codex 负责：

- 判断任务该由谁执行
- 拆分原子任务
- 编写 Kimi 可执行的清晰任务书
- 审查 Kimi 的结果
- 向用户反馈执行过程、改动范围、原因和风险

当前模式不是“把项目交给 Kimi”，而是“Codex 主导，Kimi 受控协作”。

## 任务分档

当前将任务按以下几档处理：

### 0. 非常小的任务

定义：

- 用户一眼就能确认很简单
- 用户自己顺手改更快
- 不值得额外调度 Codex 或 Kimi

处理方式：

- 由用户直接修改
- 用户会告知 Codex 该改动已经发生
- Codex 在后续分析时将其视为既有上下文，而不是重复实现

### 1. 小任务

定义：

- 改动范围很小
- 目标明确
- Codex 直接实现通常比调度 Kimi 更省 token、更快

处理方式：

- 默认由 Codex 直接完成
- 如果用户明确要求用 Kimi，或该任务虽然小但需要保留“可外包任务书”的样本，再考虑交给 Kimi

### 2. 中等任务

定义：

- 涉及多个文件，但边界清晰
- 适合拆成若干原子子任务
- 用户希望保留每一步的设计、执行和审查过程

处理方式：

- 优先由 Codex 先分析并拆分
- 再决定其中哪些原子任务交给 Kimi
- 这是当前最适合“Codex 主导 + Kimi 执行”的区间

### 3. 大任务

定义：

- 需要先进行设计、拆解、排序
- 不能直接一次性交给单个 agent 执行

处理方式：

- 先由 Codex 主导做任务拆解
- 再按原子子任务逐个推进
- 不允许把整包任务直接交给 Kimi 一口气做到底

## 默认职责分工

### 用户

- 定义目标和约束
- 决定是否接受某一步实现结果
- 决定是否进入下一步
- 决定哪些非常小的任务自己顺手修改

### Codex

- 阅读代码与文档，建立上下文
- 判断任务分档
- 决定自己做还是调用 Kimi
- 为 Kimi 编写明确任务书
- 审查 Kimi 的改动、解释原因、汇报风险
- 在实现阶段和测试阶段之间做边界控制

### Kimi

- 在受控范围内执行单个原子任务
- 不自行扩展需求
- 不自行串联实现、测试、验证等多个阶段

## 当前 Kimi 使用原则

### 1. Codex 主导

Kimi 不是主控 agent。默认由 Codex 决定：

- 这一步要不要使用 Kimi
- 这一步的目标是什么
- 这一步允许改哪些文件
- 这一步禁止做什么
- 结果是否合格

### 2. 一次只做一个原子任务

Kimi 每次只处理一个边界清晰的任务，例如：

- “只实现某个接口改动”
- “只修一个明确的 bug”
- “只补某个模块的实现代码”

不允许默认一次做：

- 实现 + 测试 + 跑测试
- 多个彼此独立的功能
- 需求不清晰时的探索性扩展

### 3. 实现和测试分开

当前明确规定：

- 写实现代码是一个阶段
- 写测试代码是另一个阶段
- 运行验证与分析结果是再后一个阶段

除非 Codex 明确下达该阶段任务，否则 Kimi 不应自行补测试、跑测试或扩展验证。

### 4. 隔离环境优先

当使用 Kimi 修改代码时，优先在隔离 worktree 或等价隔离环境中完成，以避免和用户当前工作区的未提交改动混杂。

Codex 负责：

- 选择隔离工作区
- 审核该隔离环境中的改动
- 决定是否将其引入主工作区

### 5. 执行权限受任务书约束

当前原则不是完全禁止 Kimi 执行命令，而是：

- 是否执行命令由 Codex 在任务书中明确约束
- 默认不把“写代码”和“跑验证”混成一条任务
- 如果该任务只是实现，不应顺手进行测试阶段动作

## Kimi 任务书的基本结构

Codex 在给 Kimi 下达任务时，任务书至少应包含以下内容：

1. 任务目标
2. 背景上下文
3. 本次允许修改的文件范围
4. 本次明确禁止的行为
5. 完成标准
6. 输出要求

其中“禁止事项”尤其重要，常见示例包括：

- 不要补测试
- 不要修改未列出的文件
- 不要做额外重构
- 不要扩大需求范围
- 不要运行与本任务无关的命令

## Codex 对用户的回报要求

每次调用 Kimi 后，Codex 需要尽量向用户反馈：

- 当前在做什么
- 为什么这一步交给 Kimi
- Kimi 被要求做什么
- 实际改了哪些文件
- 为什么这样改
- 有哪些风险、疑点或待确认项

目标不是堆日志，而是让用户始终知道“谁在做什么、为什么这么做、结果是什么”。

## 何时不该用 Kimi

以下情况默认不优先使用 Kimi：

- 用户自己可以顺手完成的极小改动
- Codex 自己直接完成明显更快的小任务
- 需求边界还不清楚
- 涉及高风险改动但尚未形成明确任务书
- 当前更需要设计、分析、波形对齐或故障定位，而不是实现

## 这份文档的维护规则

- 当协作规则发生变化时，优先更新本文件
- 当只是记录某次执行案例时，写入 `session-log.md`
- 当发现流程问题、优化点、潜在自动化机会时，写入 `findings-and-improvements.md`
- 等积累足够多真实样本后，再考虑提炼为 skill
