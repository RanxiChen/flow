## 2026-04-20 - 初始化协作流程

### 输入

- 用户目标：探索如何驱动本地 Kimi，把它作为可调用 agent 纳入本仓库开发流程，并保持对项目的掌控。
- 约束：
  - Codex 为主导者
  - Kimi 只做受控执行
  - 实现与测试分阶段推进
  - 每一步都要对用户可解释、可回报

### 分档判断

- 任务分档：流程设计任务
- 执行者选择：由 Codex 主导，不交给 Kimi
- 判断理由：当前先建立方法论和规则，不是实现代码阶段，直接交给 Kimi 会过早固化未成熟流程

### 执行方式

- 是否使用 Kimi：否
- 是否使用隔离 worktree：否
- 本次阶段：流程设计与文档沉淀

### 已确认规则

- 增加一档“非常小的任务”，由用户自己顺手修改并告知 Codex
- 小任务通常由 Codex 直接完成
- 中等任务最适合“Codex 主导 + Kimi 执行原子任务”
- 大任务必须先拆解，再逐个子任务推进
- Kimi 以单个原子任务模式使用
- Kimi 修改代码时优先使用隔离 worktree
- 写实现与写测试明确分开，不自动串联

### 观察

- 有效做法：
  - 先用对话把关键约束逐条定下来，再落文档
  - 先积累流程文档和真实样本，而不是立刻做成 skill
- 暴露问题：
  - 当前流程还没有经过多轮真实任务验证
  - 还没有形成稳定的 Kimi 任务书模板
- 后续改进想法：
  - 后续执行几轮真实任务后，把代表性案例补进本文档
  - 把有效的任务书结构沉淀成标准模板
