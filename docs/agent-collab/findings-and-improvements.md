# Agent Collaboration Findings And Improvements

本文档专门记录当前协作流程中的发现、问题和未来改进方向。它是后续生成 skill 的原材料之一。

## 当前发现

### 1. 不是所有任务都适合调用 Kimi

当前已经明确：

- 极小任务通常不值得调度 Kimi
- 小任务很多时候 Codex 直接实现更省 token
- 中等任务是当前最适合“Codex 主导 + Kimi 执行”的区间
- 大任务不能整包外包，必须先拆解

这说明未来 skill 里必须包含“先判断任务分档，再决定执行者”的步骤，不能默认一律调用 Kimi。

### 2. 单次 token 成本与流程收益并不线性

当前观察：

- 对小任务，Codex -> Kimi -> Codex 的往返开销占比很高
- 对中等任务，额外开销开始换来更好的分层、审查和掌控感

后续需要继续记录：

- 哪一类任务开始明显值得交给 Kimi
- 哪一类任务即使中等，也因为上下文过于紧密而不值得外包

### 3. “Codex 主导”必须被写成明确规则

仅仅说“协作”不够，当前已经形成更具体的原则：

- Codex 负责拆任务
- Codex 负责写任务书
- Codex 负责审结果
- Kimi 不是主控，不自己决定下一阶段

后续 skill 若生成，必须把这一点写成硬约束。

### 4. 实现、测试、验证分析必须拆阶段

当前已明确：

- 写实现代码是一回事
- 写测试代码是一回事
- 跑测试和分析结果又是一回事

这个拆分很重要，因为它让每一步更可控，也更方便用户审查。

## 当前问题

### 1. 还没有稳定的 Kimi 任务书模板

目前已经知道任务书至少需要：

- 目标
- 背景
- 文件范围
- 禁止事项
- 完成标准

但还没有经过足够多次任务验证，因此还不能认定是成熟模板。

### 2. 还没有稳定的“何时引回主工作区”规则

当前只明确了：

- Kimi 优先在隔离 worktree 中工作
- Codex 审完后再决定是否引回

但还缺少更细的约定，例如：

- 什么情况下直接摘取补丁
- 什么情况下只参考实现、手动重写
- 什么情况下整包放弃

### 3. 还缺少真实样本来校准任务分档

目前的分档主要来自设计判断，后续需要真实样本支撑，例如：

- 哪些任务实际证明“Codex 直接做更划算”
- 哪些任务证明“交给 Kimi 更合适”
- 哪些任务表面看是中等，实际上不适合外包

## 可实现的提升

### 1. 增加标准化任务书模板

未来可以补一份独立文档，专门存放：

- 实现阶段任务书模板
- 测试阶段任务书模板
- 验证阶段任务书模板

### 2. 增加引回主工作区的决策模板

未来可补一个检查清单，例如：

- 改动是否越界
- 文件范围是否符合预期
- 是否混入无关重构
- 是否存在用户未确认的行为变化

### 3. 增加流程回报模板

Codex 对用户的汇报目前依赖即时发挥。后续可抽象成固定格式，减少信息遗漏，例如：

- 当前阶段
- 交给谁执行
- 输入任务
- 改动结果
- 风险和待确认项

### 4. 等样本足够后再提炼为 skill

当前判断是：

- 现在可以先积累文档
- 暂时不适合直接做成 skill
- 至少经过几轮真实任务之后，再把重复模式提炼成 skill 更稳妥

## 未来需要重点观察的问题

- 哪些提示词约束最能防止 Kimi 扩需求
- 哪些任务边界定义方式最有效
- 哪些场景下隔离 worktree 的收益最大
- 实现阶段和测试阶段之间如何高效衔接
- 用户希望看到的最小必要回报粒度是什么
