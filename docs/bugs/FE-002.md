# FE-002

## Title

Frontend `s2` PC context is overwritten by a new request when miss refill response becomes visible.

## Status

open

## Symptom

During frontend recovery after an icache miss refill, the cache return becomes visible to the frontend, but the PC context latched in frontend `s2` no longer matches the PC that originally generated the returning cache data.

The observed failure pattern is:

- cache refill for the miss line starting at `0x0` completes normally
- one cycle later, cache response becomes visible to the frontend
- in that same cycle:
- `s2_respValid = true`
- but `s2_pcReg` is already a newer PC such as `0xc`
- and `s1_pcReg` has already advanced to `0x10`

This indicates that the returning cache data is being consumed while frontend pipeline PC bookkeeping has already moved forward to newer requests.

## Expected

When a miss refill response becomes visible to frontend `s2`, the PC context associated with that response must still match the original request that the cache response belongs to.

More concretely:

- the cache response payload and the frontend PC context must remain aligned
- a new request may be allowed to enter in parallel only if it does not overwrite the PC context needed by the returning response
- frontend must not associate a refill return for an older miss with a newer `s2_pcReg`

## Repro

Dedicated reproduction testcase:

- `design/src/test/scala/frontend/breezefrontendFE002Spec.scala`
- `flow.frontend.BreezeFrontendFE002Spec`

Recommended run command:

```bash
cd design
sbt "testOnly flow.frontend.BreezeFrontendFE002Spec"
```

Current testcase behavior is intentionally loose and observation-oriented:

1. `fetchBuffer.canAccept3` is held high for the whole run, so frontend always has room to continue.
2. `resetAddr` is fixed at `0x0`.
3. After reset release, the testcase does not assert cycle-by-cycle frontend expectations.
4. It continuously monitors `nextLevelReq.req`.
5. When the miss request becomes valid, it latches `nextLevelReq.paddr` and checks that it is `0x0`.
6. Starting from the cycle after observing the request, it waits a fixed `6` cycles.
7. It then returns one full 32B cache line in one cycle through `nextLevelRsp.vld/data`.
8. After that it continuously monitors the cache-to-frontend response visibility through frontend debug signals and prints:
- `cache_drsp_valid`
- `cache_drsp_vaddr`
- `s2_respValid`
- `s2_pcReg`
- `s1_pcReg`
- `s3_pcReg`

Observed miss request:

```text

[INFO] FE002 observed miss request at cycle 3, paddr=0x0
```

Observed refill injection:

```text
[INFO] FE002 send refill at cycle 9: data=0x80041300700393006003130050029300400213003001930020011300100093
```

Key observed log:

```text
[INFO] FE002 cycle 11: nextLevelReq.req=false nextLevelReq.paddr=0x0 cache_drsp_valid=true cache_drsp_vaddr=0xdeadbeef s2_respValid=true s2_pcReg=0xc s1_pcReg=0x10 s3_pcReg=0x0
[INFO] FE002 response observe at cycle 11: cache_drsp_valid=true cache_drsp_vaddr=0xdeadbeef s2_respValid=true s2_pcReg=0xc
```

This is inconsistent with the intended response context for the returning miss line that started at `0x0`.

Later cycles show that the frontend keeps rolling forward while the cache response context remains misaligned:

```text
[INFO] FE002 cycle 12: ... cache_drsp_valid=true cache_drsp_vaddr=0xdeadbeef s2_respValid=true s2_pcReg=0x10 s1_pcReg=0x14 s3_pcReg=0xc
[INFO] FE002 cycle 13: ... cache_drsp_valid=true cache_drsp_vaddr=0xdeadbeef s2_respValid=true s2_pcReg=0x14 s1_pcReg=0x18 s3_pcReg=0x10
```

This does not yet prove the final architecturally visible corruption by itself, but it does prove that:

- the frontend pipeline PC context seen at `s2` is no longer the original miss-response context when cache return becomes visible
- and the cache response `vaddr` field is currently not driven correctly either, because it remains `0xdeadbeef` instead of a real response address/context

## Current Hypothesis

The root cause is likely in frontend `s2` update priority during the cycle where cache response visibility and new request acceptance happen at the same time.

Current frontend logic:

```scala
when(icache.io.dreq.fire) {
    s2_validReg := true.B
    s2_pcReg := s1_pcReg
}.elsewhen(s2_respValid) {
    s2_validReg := false.B
}
```

If `icache.io.dreq.fire` and `s2_respValid` are both true in the same cycle, `dreq.fire` wins priority, so `s2_pcReg` is overwritten by the new request PC.

At the same time, cache is allowed to reopen request acceptance when miss handling completes, which means:

- the miss return becomes visible
- a new request is accepted
- frontend `s2` PC context may be replaced in the same cycle

That combination likely causes the response data / PC association bug.
