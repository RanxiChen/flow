# FE-002

## Title

Frontend `s2` PC context is overwritten by a new request when miss refill response becomes visible.

## Status

resolved

## Symptom

During frontend recovery after an icache miss refill, the cache return becomes visible to the frontend, but the PC context latched in frontend `s2` no longer matches the PC that originally generated the returning cache data.

The observed failure pattern is:

- cache refill for the miss line starting at `0x0` completes normally
- one cycle later, cache response becomes visible to the frontend
- in that same cycle:
- `s2_respValid = true`
- but `s2_pcReg` is already a newer PC such as `0xc`
- and `s1_pcReg` has already advanced to `0x10`

This indicates that the returning cache data is being consumed while frontend pipeline PC bookkeeping has already moved forward to newer requests.

## Expected

When a miss refill response becomes visible to frontend `s2`, the PC context associated with that response must still match the original request that the cache response belongs to.

More concretely:

- the cache response payload and the frontend PC context must remain aligned
- a new request may be allowed to enter in parallel only if it does not overwrite the PC context needed by the returning response
- frontend must not associate a refill return for an older miss with a newer `s2_pcReg`

Recommended run command:


Current testcase behavior is now tightened into a cycle-checked regression:

1. `fetchBuffer.canAccept3` is held high for the whole run.
2. `resetAddr` is fixed at `0x0`.
3. The testcase waits for the first miss request and checks that `nextLevelReq.paddr == 0x0`.
4. It waits a fixed `6` cycles after the miss request is observed.
5. It returns one full 32B cache line in one cycle through `nextLevelRsp.vld/data`.
6. It checks the first response-visible cycle:
- `cache_drsp_valid`
- `cache_drsp_vaddr`
- `s2_respValid`
- `s2_pcReg`
- `s1_pcReg`
7. It then checks the following `8` post-response cycles:
- `s3_pcReg`
- `fetchBuffer.bits.pc`
- `fetchBuffer.bits.inst`
- decoded instruction printouts for readability
- no new `nextLevelReq.req` while `0x0 ~ 0x1c` are streamed from the same cache line

At the time, this proved that:

- the frontend pipeline PC context seen at `s2` is no longer the original miss-response context when cache return becomes visible
- and the cache response `vaddr` field was also not yet wired correctly

## Final Root Cause

The final root cause was in `BreezeCache` internal `s2` miss context tracking, not in the frontend response priority itself.

The cache had been partially converted from a simple `RegNext` pipeline stage into an explicitly-latched miss context:

- `s2_vaddr` was changed to hold the miss address stably until refill completed
- but `s2_word_offset` was still implemented as `RegNext(s1_word_offset)`

That meant the miss context inside cache was split incorrectly:

- `s2_vaddr` belonged to the original miss
- `s2_word_offset` could still drift with later `s1` traffic

As a result, when refill data returned, cache could slice the returning 32B line using the wrong word offset. This made the cache response context appear inconsistent and fed misleading symptoms into frontend-side observation.

The minimal functional fix was:

- keep `s2_vaddr` as an explicitly latched miss context
- latch `s2_word_offset` in the same miss-entry event
- keep both stable until the miss refill is completed

## Resolution

The issue is considered resolved.

The fix consisted of two parts:

1. `BreezeCache` was corrected so that `s2` miss context is internally consistent:
- `s2_vaddr` and `s2_word_offset` are now latched together on miss entry
- refill data slicing now uses the correct stable word offset

2. Frontend-side tests were revalidated and updated to the corrected timing model:
- `design/src/test/scala/frontend/breezefrontendFE002Spec.scala` now checks the first response-visible cycle plus the following `8` streamed instructions from the same 32B line
- `design/src/test/scala/frontend/breezefrontendSpec.scala` was updated so its cycle expectations match the corrected cache/frontend timing
