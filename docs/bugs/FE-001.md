# FE-001

## Title

Cache cannot accept the first fetch request correctly after reset release.

## Status

fixed

## Symptom

Early observation after reset release suggested that the frontend default flow did not advance as expected. In the earlier test log, `s1_pcReg` was observed as `0` in both `cycle 0` and `cycle 1`, and this was initially interpreted as "cache did not accept the first fetch request".

## Expected

After reset release, cache should accept the first fetch request normally, the frontend default flow should advance, and `s1_pcReg` should move from `0` to `4`.

## Repro

The issue is now reproduced and analyzed through the dedicated testcase:

- `design/src/test/scala/frontend/breezefrontendSpec.scala`
- `BreezeFrontendFE001Spec`
- `"BreezeFrontend FE-001 bug replication scenario"`

Run:

```bash
cd design
sbt 'testOnly flow.frontend.BreezeFrontendFE001Spec'
```

The current testcase keeps the `cycle 0` and `cycle 1` debug prints and checks:

- `cycle 0`: cache `s0` is ready and the first request fires
- `cycle 1`: cache `s0` is not ready, and cache `s1` holds a miss

## Root Cause

The original interpretation was wrong because frontend and cache pipeline observation after reset were not aligned.

- What was observed as "no handshake yet" at the frontend boundary was already one cycle after the cache had accepted the first request
- After fixing cache valid initialization in simulation, the trace shows:
- `cycle 0`: cache `s0` is `ready` and `fire`
- `cycle 1`: the accepted request has already advanced into cache `s1`, and `s1_hit` is `false`
- At that point cache blocks new `s0` traffic, so the later observation of `no ready` is the miss-handling behavior after the first request has already entered the cache

So the issue was not that the first request failed to enter the cache. The real problem was that reset-phase behavior and observation points between frontend and cache were not aligned, which led to an incorrect bug conclusion.

## Resolution

This bug is considered fixed as a debug/reproduction issue.

- No RTL source behavior was changed to close FE-001
- The fix was done in simulation/test code only
- `BreezeCache` valid initialization was made explicit in simulation (`s1_valid` starts from `false`)
- The FE-001 testcase was reduced and instrumented so that the reset boundary and cache pipeline state are observed correctly
